# Интеграция Keycloak (портал EstiMat) с Active Directory — инструкция для админов компании

Документ для **сетевого отдела и администраторов Active Directory компании**. Описывает, что
нужно настроить на стороне компании, чтобы сотрудники входили в корпоративные порталы своими
доменными учётными записями.

**Как это работает.** На стороне поставщика портала развёрнут Keycloak (корпоративный сервис
аутентификации) на сервере с публичным адресом. Сотрудник вводит доменный логин и пароль на
странице входа; Keycloak проверяет их в вашем Active Directory по защищённому каналу и **не
хранит у себя пароли сотрудников**. Подрядчики в AD не заводятся — они ведутся отдельно в
Keycloak.

Связь Keycloak ↔ контроллер домена идёт **только через site-to-site IPsec-туннель** по протоколу
**LDAPS (порт 636/TCP)**. Контроллер домена в интернет не публикуется.

```
Сотрудник ──HTTPS──> Keycloak (auth.su10.ru, публичный сервер)
                        │
                        │  LDAPS 636/TCP
                        ▼
                  site-to-site IPsec (IKEv2)
                        │
                        ▼
              Контроллер домена (внутренняя сеть компании)
```

---

## 0. Обмен данными перед началом (заполнить совместно)

| Параметр | Кто предоставляет | Значение |
|---|---|---|
| Публичный IP сервера Keycloak (наша сторона) | Поставщик портала | `<VPS_PUBLIC_IP>` |
| Публичный IP вашего VPN-шлюза | Компания | |
| Внутренние IP / FQDN контроллеров домена (DC) | Компания | |
| Подсети или точные `/32` адреса DC для туннеля | Компания | |
| Способ аутентификации IPsec (PSK или сертификаты) | Совместно | |
| PSK / сертификаты IPsec | Совместно, **защищённым каналом** | |
| Base DN домена | Компания | напр. `DC=corp,DC=ru` |
| Users DN (OU с сотрудниками) | Компания | |
| Groups DN (OU с группами) | Компания | |
| Login-атрибут | Компания | `sAMAccountName` или `userPrincipalName` |
| Bind DN сервисной учётки | Компания | напр. `CN=svc-keycloak-ldap,OU=Service,DC=corp,DC=ru` |
| Пароль сервисной учётки | Компания, **защищённым каналом** | |
| Публичная часть корпоративного CA (root/intermediate) | Компания | файл `.pem`/`.cer` |

> Все секреты (PSK, пароль bind-учётки, приватные ключи) передаются **только защищённым
> каналом** (корпоративный менеджер секретов, зашифрованный архив с паролем по другому каналу).
> **Не** по электронной почте и не в мессенджерах открытым текстом. **Приватные ключи
> сертификатов DC не передаются никогда** — нужна только публичная часть CA.

## 1. Сеть: site-to-site IPsec (IKEv2)

1. Поднять на корпоративном VPN-шлюзе (Cisco / MikroTik / pfSense / Fortinet / strongSwan и т.п.)
   **site-to-site IPsec, IKEv2** навстречу публичному IP нашего сервера (`<VPS_PUBLIC_IP>`).
2. Согласовать параметры (рекомендуемый набор, уточняется при настройке):
   - IKEv2; аутентификация — **PSK** или **сертификаты**;
   - шифрование — современное: **AES-256-GCM**, целостность/PRF — **SHA-256**, группа DH — **14
     и выше** (например 19/20 на эллиптике, если поддерживается обеими сторонами);
   - время жизни SA — по вашей политике.
3. **Узкие traffic selectors:** разрешить в туннеле трафик только между нашим сервером
   (`<VPS_PUBLIC_IP>/32`) и конкретными контроллерами домена (`<DC_IP>/32`). Всю внутреннюю сеть
   в туннель **не** заворачивать.
4. **Firewall:** от нашего сервера к DC разрешить **только 636/TCP (LDAPS)**. Порт `3269/TCP`
   (Global Catalog over SSL) — только если потребуется поиск по глобальному каталогу. Остальное
   закрыть.
5. Настроить на своей стороне **мониторинг и алерты** доступности туннеля и порта LDAPS —
   простой канала означает невозможность входа сотрудников.

---

## 2. LDAPS на контроллерах домена

1. Включить **LDAPS** на контроллерах домена (по умолчанию AD отдаёт LDAP; для LDAPS нужен
   установленный серверный сертификат).
2. Серверный сертификат DC должен:
   - быть выпущен корпоративным/доверенным CA;
   - содержать в **SAN** имя, по которому Keycloak обращается к DC (FQDN или IP);
   - иметь EKU **Server Authentication** (`1.3.6.1.5.1.5.7.3.1`).
3. Убедиться, что DC слушает **636/TCP** и LDAPS-подключение проходит (например, `ldp.exe` →
   Connection → SSL, порт 636).
4. Передать нам **публичную часть** корневого/промежуточного **CA** (`.pem`/`.cer`) — мы добавим
   её в доверенные на стороне Keycloak. Приватные ключи **не** передаются.

---

## 3. Сервисная учётная запись для LDAP bind

1. Создать сервисную учётную запись, например `svc-keycloak-ldap`.
2. Права — **минимальные, только чтение (read)**:
   - доступ на чтение к OU с сотрудниками (Users DN) и к OU с группами (Groups DN);
   - запрет интерактивного входа (logon);
   - **не** включать в административные группы (Domain Admins и т.п.);
   - пароль не истекает, либо согласовать процедуру ротации с уведомлением нас заранее.
3. Keycloak работает с каталогом в режиме **READ-ONLY**: пользователи и пароли остаются в AD,
   Keycloak их не изменяет.
4. Передать нам **Bind DN** (полный DN учётки) и **пароль** (защищённым каналом).

---

## 4. Группы доступа и ролей в AD

Доступ к порталу и роли определяются членством в группах AD. Создайте группы (имена — пример,
согласуйте свои) и сообщите их **полные DN**:

| Группа AD (пример) | Назначение в портале EstiMat |
|---|---|
| `EstiMat-Access` | Общий доступ к порталу. **Без членства в ней сотрудник в портал не входит.** |
| `EstiMat-Admins` | Роль «Администратор» |
| `EstiMat-Engineers` | Роль «Инженер-сметчик» |
| `EstiMat-Managers` | Роль «Руководитель» |

Правила:
- сотруднику для входа нужна как минимум `EstiMat-Access`; роль выдаётся отдельными группами;
- роль «Подрядчик» сотрудникам компании **не** назначается — она только для внешних подрядчиков
  (ведутся в Keycloak отдельно);
- членство кэшируется; при изменении состава групп доступ обновляется при следующем входе/синхронизации.

---

## 5. Учёт состояния учётных записей

- **Отключённые/заблокированные** в AD учётки (`userAccountControl`: disabled / locked) должны
  терять доступ к порталу. Keycloak это учитывает — отдельных действий не требуется, но при
  увольнении/блокировке сотрудника достаточно стандартного отключения учётки в AD.
- Смена доменного пароля действует сразу — портал отдельный пароль не хранит.

---

## 6. Что делает наша сторона (для понимания границы ответственности)

- Поднимаем IPsec-туннель навстречу вашему шлюзу, проверяем доступность DC по 636/TCP.
- Импортируем ваш CA в доверенные, настраиваем в Keycloak LDAPS-федерацию (READ-ONLY) с вашей
  сервисной учёткой и параметрами каталога.
- Сопоставляем ваши группы AD с ролями портала (раздел 4).
- Проверяем входы и мониторим bind/туннель со своей стороны.

---

## 7. Приёмочная проверка (совместно)

1. Туннель IPsec поднят, стабилен; с нашего сервера проходит LDAPS bind к DC.
2. Сотрудник — член `EstiMat-Access` — успешно входит в портал.
3. Сотрудник **без** `EstiMat-Access` — входа в портал не получает.
4. Отключённая в AD учётка — входа не получает.
5. Членство в `EstiMat-Admins/Engineers/Managers` корректно даёт соответствующую роль.

---

## Сводка: что прислать нашей стороне

1. Публичный IP VPN-шлюза, IP/FQDN контроллеров домена, подсети/`/32` для туннеля.
2. Параметры IPsec и PSK/сертификаты (защищённым каналом).
3. Публичную часть корпоративного CA.
4. Bind DN и пароль сервисной учётки `svc-keycloak-ldap` (защищённым каналом).
5. Base DN, Users DN, Groups DN, login-атрибут.
6. DN групп `EstiMat-Access`, `EstiMat-Admins`, `EstiMat-Engineers`, `EstiMat-Managers`.
