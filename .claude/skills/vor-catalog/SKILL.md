# vor-catalog — справочник работ/материалов из ВОР

Цель: качественный, **компактный** корпоративный справочник работ — без похожих/дублирующих работ. Синонимы сливаются в одну типовую работу с каноническим наименованием, исходные формулировки сохраняются как алиасы. Для каждой работы — набор типовых материалов с коэффициентом расхода на единицу работы.

Сбор и проверка данных — работой **агентов** (Workflow), не типовыми скриптами. Python/openpyxl используется только как «руки» для чтения ячеек.

## Файлы

| Путь | Назначение |
|---|---|
| `temp/ВОР ЭОМ/*.xlsx` (или новая папка) | входные ВОР |
| `temp/справочник/raw/*.json` | механические дампы непустых ячеек (`temp/dump_raw.py`) |
| `temp/справочник/work/extracted_*.json` | результат экстракторов (по файлу) |
| `temp/справочник/catalog.json` | канонический каталог + журнал контроля |
| `temp/справочник/works.md` | основной справочник для человека |
| `temp/справочник/works_raw.md` | все исходные формулировки → типовая работа |
| `temp/справочник/db_snapshot.json` | дамп справочника БД smeta (`temp/dump_db.cjs`, read-only) |
| `temp/справочник/mapping.json|.md` | соответствия каталог ↔ БД |
| `scripts/vor/rules.json` | **накапливаемые правила** (синонимы, не-синонимы, маппинг разделов, уроки) |

## Процесс для новых ВОР

1. Положить файлы в папку входных ВОР.
2. Механический дамп: адаптировать/запустить `temp/dump_raw.py` (лист «ВОР», все непустые ячейки → JSON).
3. Запустить Workflow **vor-catalog-build** (шаблон: `вор-workflow-build.md` рядом с этим скиллом, либо последний скрипт в сессии): фазы Извлечение (агент на файл) → Каталогизация (слияние с существующим `catalog.json`!) → Контроль (адверсариальный, цикл исправлений ≤3).
4. Просмотреть `notes`/журнал контроля: новые синонимы и спорные слияния показать пользователю.
5. Подтверждённые решения занести в `scripts/vor/rules.json` (`synonymGroups` / `notSynonyms` / `lessons`).
6. Дамп БД (`temp/dump_db.cjs`) → Workflow **vor-catalog-match**: БД-маппер → контроллер маппинга → `mapping.json`.
7. Импорт: пользователь на машине с write-доступом: `npm run db:migrate -w server`, затем `npm run db:import-vor -w server -- temp/справочник/catalog.json temp/справочник/mapping.json`.

## Роли агентов (промпты — по образцу workflow-скриптов прошлых прогонов)

- **Экстрактор** (×1 на файл): сам интерпретирует структуру листа (шапка, разделы «Подзаголовок…», пары Р→мат., объёмы/расходы); не теряет ни одной строки; аномалии — в отчёт.
- **Каталогизатор**: сливает синонимы семантически (как инженер-сметчик), формирует канонические имена (без дат-меток прайса и проектных суффиксов), классифицирует по `sectionToCostType` из rules.json, считает qty_ratio (медиана consumption/volume).
- **Контроллер** (отдельный контекст, адверсариальный): счётчики Р/мат. против сырых дампов; полнота алиасов; отсутствие похожих работ и ошибочных слияний; согласованность ед.изм.; выборочный пересчёт коэффициентов; осмысленность классификации.
- **БД-маппер + контроллер маппинга**: семантические соответствия с расценками БД (категория «ВИС / Электрические системы»), без ложных совпадений.

## Зафиксированные знания

- БД `smeta` (Yandex, кластер `c-c9qmbgvs6rit4qfe0dni`): чтение через MCP `postgres_smeta` (user `mcp_readonly`). Электрика — категория «ВИС / Электрические системы» (8 видов). Демо-категории «Электромонтажные работы»/«Общестроительные работы» — не цель импорта.
- Связь «работа → типовые материалы» — таблица `rate_materials` (qty_ratio = расход на ед. работы). При добавлении работы в смету по расценке сервер автоматически добавляет материалы со `status='suggested'` («предложение», кнопки ✓/✗ в UI).

## Уроки (пополнять после каждого прогона!)

- Формат ВОР: шапка в строках 5–7; в Зил33 есть строка нумерации колонок («1»,«2»,…) — не данные. Колонки: 2=КОД (Р/мат.), 3=наименование, 4=ед., 5=объём работы, 6=расход материала.
- Строки корпусов/шифров РД («10.03Б…Корпус 4») — не разделы и не работы.
- Проектные суффиксы материалов (ВРУ-К4/К5, ЩКК1..6, «на 3/4/6 квартир») — один типовой материал.
- Передача кириллических python-скриптов через stdin в PowerShell ломает кодировку — писать скрипт во временный файл с `-Encoding utf8`.
- `claude mcp add` доступен; новый MCP-инструмент появляется только после перезапуска сессии — до этого читать БД через node pg со строкой подключения из `.claude.json`.
