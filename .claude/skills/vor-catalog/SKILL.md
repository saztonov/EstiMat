# vor-catalog — справочник работ/материалов из ВОР

Цель: качественный, **компактный** корпоративный справочник работ — без похожих/дублирующих работ. Синонимы сливаются в одну типовую работу с каноническим наименованием, исходные формулировки сохраняются как алиасы. Для каждой работы — набор типовых материалов с коэффициентом расхода на единицу работы.

Сбор и проверка данных — работой **агентов** (Workflow), не типовыми скриптами. Python/openpyxl используется только как «руки» для чтения ячеек.

## Файлы

| Путь | Назначение |
|---|---|
| `temp/ВОР ЭОМ/*.xlsx` (или новая папка) | входные ВОР |
| `temp/справочник/raw/*.json` | механические дампы непустых ячеек (`temp/dump_raw.py`) |
| `temp/справочник/work/extracted_*.json` | результат экстракторов (по файлу) |
| `temp/справочник/catalog.json` | канонический каталог + журнал контроля |
| `temp/справочник/works.md` | основной справочник для человека |
| `temp/справочник/works_raw.md` | все исходные формулировки → типовая работа |
| `temp/справочник/db_snapshot.json` | дамп справочника БД smeta (`temp/dump_db.cjs`, read-only) |
| `temp/справочник/mapping.json|.md` | соответствия каталог ↔ БД |
| `scripts/vor/rules.json` | **накапливаемые правила** (синонимы, не-синонимы, маппинг разделов, уроки) |

## Процесс для новых ВОР (отработан на 29 уникальных ВОР, 5 проектов)

1. Положить файлы в папку входных ВОР (структура «Проект/.../файл.xlsx»).
2. Механический дамп: `temp/dump_raw2.py` (рекурсивно, дедуп по SHA256, выход `raw2/<Проект>__<имя>.json` + `_manifest.json`). Копии с микроправками SHA не ловит — проверь хеш содержимого строк (`temp/dedup2.py`) и состав работ; копию исключить из расчётов постобработкой.
3. Workflow **vor-catalog-build-v2** (скрипт сохранён в сессии; паттерн): на проект — конвейер «извлечение (агент на файл) → проектный каталогизатор (внутрипроектная канонизация; пары хранят ПОФАЙЛОВЫЕ ratio) → проектный контроллер (цикл ≤2)»; затем сводный агент строит `merge_map.json` и МЕХАНИЧЕСКИ собирает `catalog_v2.json` своим python-скриптом (assemble; агент решает только слияния); затем сводный контроллер (цикл ≤3): покрытие биективно, все мультипроектные слияния, единицы, сплошной пересчёт ratioMedian, классификация.
4. Постобработка (`temp/postprocess_v2.py`): дедуп копий, пересчёт медиан, расчёт `isTypical` (см. ниже), notes по остаточным замечаниям. Обзор: `temp/gen_works_md.py` → `works_v2.md`.
5. Просмотреть notes/журналы; решения занести в `scripts/vor/rules.json`.
6. Маппинг: Workflow **vor-catalog-match-v2** — входы по группам видов (`temp/prep_match.py` → `work2/match_input_*.json`), 5 мапперов параллельно → контроллер (полнота 168/168, дубли rateId между частями, ложные matched) → части `work2/mapping_part_*.json` → сборка `python scripts/vor/assemble_mapping.py` → `mapping_v2.json` (84 matched / 71 probable / 13 toCreate).
7. Excel для анализа: `python scripts/vor/gen_excel_v2.py` → `Справочник_ЭОМ_v2.xlsx` (листы Работы / Материалы к работам / Материалы / Источники / Легенда).
8. Импорт (пользователь, машина с write-доступом): `npm run db:migrate -w server`, затем `npm run db:import-vor -w server -- temp/справочник/catalog_v2.json temp/справочник/mapping_v2.json`. Импорт пишет ТОЛЬКО в новый справочник (rates_v2/materials_v2/rate_materials_v2; миграция 0010), старый справочник не изменяется; в rate_materials_v2 попадают только типовые пары.

## Типовость (правило пользователя, 2026-06-11)

`isTypical = pairProjects/workProjects > 0.5 && pairFiles/workFiles > 0.5 && pairFiles >= 2`. Редкие/уникальные материалы в БД НЕ заносятся (только в Excel). На 29 ВОР: из 1049 пар типовых 104 (75 уникальных материалов у 41 работы).

## Роли агентов (промпты — по образцу workflow-скриптов прошлых прогонов)

- **Экстрактор** (×1 на файл): сам интерпретирует структуру листа (шапка, разделы «Подзаголовок…», пары Р→мат., объёмы/расходы); не теряет ни одной строки; аномалии — в отчёт.
- **Каталогизатор**: сливает синонимы семантически (как инженер-сметчик), формирует канонические имена (без дат-меток прайса и проектных суффиксов), классифицирует по `sectionToCostType` из rules.json, считает qty_ratio (медиана consumption/volume).
- **Контроллер** (отдельный контекст, адверсариальный): счётчики Р/мат. против сырых дампов; полнота алиасов; отсутствие похожих работ и ошибочных слияний; согласованность ед.изм.; выборочный пересчёт коэффициентов; осмысленность классификации.
- **БД-маппер + контроллер маппинга**: семантические соответствия с расценками БД (категория «ВИС / Электрические системы»), без ложных совпадений.

## Зафиксированные знания

- БД `smeta` (Yandex, кластер `c-c9qmbgvs6rit4qfe0dni`): чтение через MCP `postgres_smeta` (user `mcp_readonly`). Электрика — категория «ВИС / Электрические системы» (8 видов); ОЗДС — слаботочная категория (своего вида нет). Демо-категории «Электромонтажные работы»/«Общестроительные работы» — не цель маппинга.
- **Новый справочник (v2)** — таблицы `rates_v2` / `materials_v2` / `rate_materials_v2` (миграция 0010): иерархия — FK на существующие `cost_types`; `legacy_rate_id` + `match_kind` (matched/probable) — ссылка на старую расценку. Старый справочник (`rates`, `material_catalog`) при импорте НЕ изменяется.
- Вкладка «Сравнение справочников» (Справочники → `CatalogComparePanel`): слева старое дерево (`GET /api/rates/tree`), справа новое (`GET /api/rates-v2/tree`); клик по работе справа → типовые материалы (`GET /api/rates-v2/:id/materials`) + подсветка legacy-расценки слева.
- Смета работает только со старым справочником, материалы добавляются вручную; механизм «предложений» (`rate_materials` + `estimate_materials.status`, миграция 0009) — спящий до перевода смет на новый справочник.

## Уроки (пополнять после каждого прогона!)

- Формат ВОР: шапка в строках 5–7; в Зил33 есть строка нумерации колонок («1»,«2»,…) — не данные. Колонки: 2=КОД (Р/мат.), 3=наименование, 4=ед., 5=объём работы, 6=расход материала.
- Строки корпусов/шифров РД («10.03Б…Корпус 4») — не разделы и не работы.
- Проектные суффиксы материалов (ВРУ-К4/К5, ЩКК1..6, «на 3/4/6 квартир») — один типовой материал.
- Передача кириллических python-скриптов через stdin в PowerShell ломает кодировку — писать скрипт во временный файл с `-Encoding utf8`.
- `claude mcp add` доступен; новый MCP-инструмент появляется только после перезапуска сессии — до этого читать БД через node pg со строкой подключения из `.claude.json`.
