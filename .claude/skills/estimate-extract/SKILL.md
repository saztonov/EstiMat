# estimate-extract — извлечение работ/материалов из РД в смету

Агент проходит по рабочей документации (РД, распознанный markdown из портала
RDLOCAL или загруженный .md) и добавляет извлечённые **работы** и **материалы** в
смету, сопоставляя их со справочником (rates/rates_v2, material_catalog/materials_v2).

Ключевой принцип — **щадящий по токенам**: основную массу делает механический
rule-based раннер (0 токенов), LLM-агент включается только на неоднозначное.

## Когда использовать
- «извлеки работы из РД <документ>», «распознай спецификацию и добавь в смету»
- «добавь материалы в смету из этого markdown»
- «подбери из справочника работы/материалы по задаче …»

## Архитектура
Ядро извлечения — чистые модули `server/src/lib/extract/` (парсер markdown →
классификатор таблиц → rule-based экстрактор → matcher → pipeline → apply).
Раннер `scripts/ai-extract/run.ts` оркеструет механику и применяет результат.

| Файл | Назначение |
|---|---|
| `scripts/ai-extract/run.ts` | CLI-раннер: rule-based прогон + автоприменение в смету |
| `scripts/ai-extract/rules.json` | накопленные правила (columnAliases, sectionToWork, unitAliases, materialSynonyms, lessons) |
| `server/src/lib/extract/pipeline.ts` | оркестратор ядра (без БД/LLM) |
| `server/src/lib/extract/matcher.ts` | сопоставление со справочником (exact/alias/fuzzy/LLM) |
| `temp/ai-extract/<jobId>.md` | markdown документа РД (fallback-вход для раннера) |

Задания хранятся в таблице `ai_jobs`; смета пишется в `estimate_items` +
`estimate_materials` со `source='ai'`. Несопоставленные позиции помечаются
`needs_review=true` и видны в смете под фильтром «Не согласованные».

## Область подбора работ (section-scope) и подбор работ

Сметчик в панели ИИ выбирает **разделы** (`cost_categories`) и опционально **виды**
(`cost_types`). Область кладётся в `ai_jobs.input.sectionScope = {categoryIds, costTypeIds}`
и сужает справочник расценок (`loadCatalogSnapshot(db, mode, scope)` — фильтр только по
работам, материалы не сужаются). РД **не содержит монтажных работ** — они подбираются из
суженного справочника:
- встроенный движок (фаза 2) вызывает `LlmPort.suggestWorks(documentDigest, candidates, ctx)`
  — выбирает работы из scope-кандидатов по содержанию документа, все с `needs_review=true`;
- агент skill (фаза 1) делает тот же подбор вручную из переданного суженного списка расценок.

Материалы извлекаются из спецификаций: GFM-таблицы (rule-based) + распознанный текст
блоков-изображений (поле «Текст на чертеже» → `isDrawingText`, разбирается LLM с отдельным
промптом строчных спецификаций; без LLM — anomaly). `normalizeTableHeader` находит настоящую
шапку (она часто не в первой строке) и отбрасывает строку нумерации колонок «1..9».

Модель LLM задаётся в Администрирование → Настройки (`ai_models` + `ai_model_default`).
Привязка материала к работе и подтверждение материала снимают `needs_review`.

## Процесс

### Фаза 0 — вход
1. Получи `jobId` (создаётся из UI кнопкой «Извлечь» → `POST /api/ai/jobs`,
   статус `pending`). Если работаешь напрямую — найди pending-задание:
   `SELECT id, estimate_id, source_kind, input FROM ai_jobs WHERE status='pending'`.
2. Обеспечь markdown:
   - `upload_md`: уже в `ai_jobs.input.markdown`.
   - `rd_document`: возьми markdown документа (`GET /api/rd/documents/:nodeId/markdown`
     при запущенном бэкенде, либо через MCP портала) и положи в
     `temp/ai-extract/<jobId>.md`.

### Фаза 1 — механика (0 токенов)
Запусти раннер:
```
npm run ai:extract -w server -- <jobId>
```
Он распарсит markdown, извлечёт структурные спец-таблицы, сопоставит со
справочником, **применит** работы/материалы в смету и напечатает отчёт:
`stats`, `anomalies`, список несопоставленных/на-проверку.

### Фаза 2 — экстрактор (LLM, только при необходимости)
Если в отчёте есть `anomalies` про неразобранные таблицы/прозу — разбери их сам:
прочитай соответствующие фрагменты markdown, извлеки позиции (наименование,
кол-во, ед.изм., маркировка) **строго по документу, без выдумок**. Зафиксируй
повторяющиеся шаблоны заголовков колонок в `rules.json → columnAliases`, а
соответствия разделов работам — в `sectionToWork`, чтобы следующий прогон сделал
это rule-based.

### Фаза 3 — матчер (LLM, для несопоставленного)
Для позиций `needs_review` без привязки подбери лучшую запись справочника из
кандидатов (по смыслу, без ложных совпадений). Обнови привязку/цену в смете
(`estimate_items`/`estimate_materials`) либо предложи правки пользователю.

### Фаза 4 — адверсариальный контроль (отдельный проход, ≤3 цикла)
Сверь, заняв скептическую позицию:
- число извлечённых позиций против количества строк в сырых таблицах (полнота);
- согласованность единиц измерения с количествами;
- отсутствие выдуманных привязок (каждая `matched` подтверждается документом);
- что `needs_review` проставлен везде, где привязка неуверенная;
- выборочно сверь `source_snippet` с исходным markdown.
Найденные расхождения исправь и повтори контроль.

### Фаза 5 — уроки
Запиши в `rules.json → lessons` новые ловушки формата РД, расширь `sectionToWork`,
`columnAliases`, `materialSynonyms`. Это снижает расход токенов на будущих прогонах.

## Правила
- Не теряй ни одной строки спецификации молча — всё в смету или в anomalies.
- Цена берётся из справочника, выбранного настройкой `ai_catalog_source`
  (Администрирование → Настройки). Не придумывай цены.
- Применение идёт автоматически (без подтверждения) — поэтому при сомнении
  ставь `needs_review`, а не выдумывай привязку.
